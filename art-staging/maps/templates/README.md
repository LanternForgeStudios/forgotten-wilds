# Tiled map templates

Four starter maps, one per map type this game has, meant to be opened directly in the
[Tiled editor](https://www.mapeditor.org/) and edited/exported from there rather than hand-typed.
Full authoring rules (layer names, required custom properties, multi-tileset support, export
settings) live in `docs/Tiled-Map-Authoring.md` — this file only covers what's specific to these
four templates. They were generated by `scripts/genMapRicher.mjs` from the specs in
`scripts/map-specs-richer/template-*.json`, but the output files themselves are what matters —
genuinely Tiled-compliant JSON, no different in shape from a real Tiled export.

| Template | Loosely based on | Dimensions | Tilesets |
|---|---|---|---|
| `building-interior.json` | `ash-hallow-elias-house.json` | 7×6 | `tiny-dungeon` + `velmora-environments` |
| `town.json` | `ash-hallow.json` | 22×14 | `ground-tiles` + `trees-signs-rocks-bridge` |
| `overworld-region.json` | `ironwood-trail.json` | 32×20 | `ground-tiles` + `trees-signs-rocks-bridge` |
| `dungeon-floor.json` | `hollow-rail-mine.json` | 30×18 | `tiny-dungeon` only |

Each demonstrates the layer model documented in `docs/Tiled-Map-Authoring.md`:
`ground` → `decorations-1` (→ `decorations-2`, ... if you need more) → `collisions` (object layer)
→ `objects` (object layer) → `overhang`. Every shipped map now uses this same model for real (see
`public/assets/maps/*.json`) — these templates are just small, illustrative versions of it.

Each template uses a single, unsuffixed `overhang` layer, which stays valid indefinitely (there's
no need to rename it). If a map needs more than one overhang layer stacked (e.g. a tree canopy
above a lower roof), the loader also supports numbered `overhang-1`, `overhang-2`, ... — same
convention as `decorations-N` — see `docs/Tiled-Map-Authoring.md`'s layer table.

## Multiple tilesets

`town.json` and `overworld-region.json` each embed **two** tilesets (a ground pack +
a separate prop pack) rather than borrowing decoration tiles from within one image - real,
standard Tiled multi-tileset usage, and how the actual shipped town/overworld maps work now too.
Each embedded tileset carries its own `tilesetAssetId` custom property (Tileset Editor →
Properties) rather than relying on the map-level one, which only the *first* tileset needs (for
backward compatibility with maps authored before multi-tileset support existed). See
`docs/Tiled-Map-Authoring.md`'s "Multiple tilesets per map" section.

## Walk-in zones

`overworld-region.json` includes one `zone` object (`walk-in-subarea-example`) alongside the usual
point objects - a rectangle the player just walks into rather than an `interactable` they approach
and press Interact on. See `docs/Tiled-Map-Authoring.md`'s `objects` layer section and
`src/hooks/useLocationExploration.ts`'s `onZoneEnter` option for how a real one (e.g. Ironwood
Trail's Hunter's Camp) is wired up.

## What's a faithful copy vs. a trimmed sample

- **Layout and object types** (doors, NPCs, shrines, chests, spawn points, transitions, the one
  `zone` example) are close to their source map, so each template still reads as "a real
  town/dungeon/etc.", not a generic box.
- **`decorations-1`/`overhang` tile ids** are a small, real sample of each tileset's actual prop/
  wall/floor variety, not exhaustive - add as many as your real map needs, following the same
  pattern.
- **The `collisions` rectangle** in each file is a single placeholder obstacle (a table, market
  stall, boulder, or cave-in) sized 1-2 tiles, just to show the mechanism. Add as many as your real
  map needs.

## Map-level metadata

Beyond the tileset block below, each file also carries the top-level fields a real Tiled install
needs just to recognize and open the file at all: `type: "map"`, `orientation`, `renderorder`,
`infinite`, `compressionlevel`, `tiledversion`, `version`, `nextlayerid`, `nextobjectid`, plus a
unique `id`/`x`/`y` on every layer and `draworder` on every object layer. The game's own loader
(`src/assets/tiledLoader.ts`) ignores all of these too - they're required by Tiled, not by this
game.

## The tileset block

Unlike a truly minimal map — where the loader only reads `firstgid`/`tilecount`/`columns`/
`tiles[].properties`/the per-tileset `tilesetAssetId` property — each template's embedded
tileset(s) also include `name`, `image`, `imagewidth`, `imageheight`, `tilewidth`, `tileheight`,
`margin`, and `spacing`. These extra fields are what a real Tiled install needs to actually open
the file and render the tileset preview; the game's loader ignores all of them.

**The `image` path (e.g. `../../../public/assets/tilesets/tiny-dungeon.png`) is for Tiled's own
local preview only.** The game never reads it — asset resolution happens entirely through each
tileset's own `tilesetAssetId` custom property. If Tiled can't find a PNG at that relative path on
your machine, either move/symlink the file to match, or just repoint the tileset's image path
inside Tiled — it has no effect on how the map loads in-game.

## Where to put a real map made from one of these

Once you've edited a template into an actual location, export/save it as
`public/assets/maps/<location-id>.json` and add a matching entry to `src/assets/registry.ts` (see
any existing `map.*` entry for the shape) — same as every other map in this repo.
